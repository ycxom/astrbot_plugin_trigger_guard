# astrbot_plugin_trigger_guard

群聊触发控制器。纯插件实现，不修改 AstrBot 核心代码。

## 解决的问题

AstrBot 内置的群聊主动回复只有一个全局概率（`provider_ltm_settings.active_reply.possibility_reply`，
0.0-1.0），写在配置文件里，无法按群、按用户单独设置；而且"每条消息独立掷骰子"这种伯努利
概率模型很反直觉——同一个 0.02 的概率，连续几百条消息里完全可能连续触发好几次，也可能
很长时间不触发一次，波动大、难预期。同时也没有"拉黑"和"完全屏蔽"的区分，唤醒方式也只
支持前缀 / @ / 引用。本插件在"请求 LLM 前"统一拦截，补上这几块能力：

- **触发区间而不是概率**：不再是"每条消息 X% 概率触发"，而是给每个作用域配置一个
  `[最少, 最多]` 条消息的区间（默认 50~100），攒够随机抽到的目标条数普通消息后触发一次，
  然后为下一轮重新抽一个目标值。更符合"这个群大概每 50~100 条消息主动说一次话"的直觉，
  波动也比纯概率模型小。
- **独立的群 / 用户触发区间**：本群用户区间 > 全局用户区间（跨群，见下）> 群区间覆盖 >
  全局默认区间。
- **不会自我触发的计数器**：机器人自己发的消息、拉黑/完全屏蔽名单里用户发的消息，都不会
  计入任何作用域的触发计数——这不只是为了"准确"，区间设得很小（比如 min=max=1）时，如果
  机器人自己的消息也被算成一条"普通消息"，主动回复发出去的那条消息会立刻又把计数器打满，
  自己跟自己无限触发下去。
- **主动回复白名单（群级别）**：`active_reply_enable` 打开后，**默认不会对任何群主动回复**，
  只有加进"主动回复白名单"的群才会跑触发区间计数——这是刻意选择的白名单模式（不是黑名单
  排除模式），避免开关一开就对机器人所在的所有群生效。
- **拉黑**：保留该用户的聊天记录（会正常进入群聊上下文感知），但该用户**不能触发 LLM**
  （无论是直接 @ / 前缀唤醒，还是被计数判定触发的主动回复，都会在请求 LLM 前被拦截）。
- **完全屏蔽**：该用户的消息**不会进入任何上下文**（直接终止事件传播，群聊感知等
  后续插件根本看不到这条消息），同时也不能触发 LLM。
- **关键词包含唤醒**：消息中包含配置的关键词即可唤醒，不要求前缀或 @。
- **消息字数上限**：按群 / 按用户设置单条消息最大字数（用户设置优先级高于群设置），
  超过上限的消息按"完全屏蔽"处理——不进上下文、不触发 LLM，**连直接 @ 唤醒也拦**，
  防止一条超长消息把持久化的人设/上下文冲淡或污染。
- **机器人配置路由管理**：给指定的群单独绑定一份"机器人配置"（人设/Provider 等），
  比 AstrBot 官方"机器人"页面里那张表更直接——选群、选配置，立即生效。跟官方页面读写
  的是同一份数据，两边改了对方都能看到。

## 配置

`_conf_schema.json` 里的开关走 AstrBot 面板（插件 -> TriggerGuard -> 配置）：

- `enable`：总开关。
- `active_reply_enable`：是否启用本插件自己的主动回复判定。**开启后会强制接管**——
  见下面"强制接管内置主动回复"。
- `global_trigger_interval_min` / `global_trigger_interval_max`：群/用户没有单独设置
  区间时使用的默认 `[最少, 最多]`（默认 50~100），`max` 必须 >= `min`。
- `debug_log`：详细日志/调试模式，默认关闭。打开后，每条群消息的触发判定过程——是否
  命中黑/白名单、是否已经是 wake 状态、`active_reply_enable` 是否开着、当前 scope 的
  计数/阈值——都会以 **INFO** 级别打印，不需要去改 AstrBot 全局日志级别，排查"配了但
  不触发"直接看日志就行。真正触发主动回复、命中黑/屏蔽名单这几类低频事件不受这个开关
  影响，始终可见。排查完建议关掉，不然大群会刷屏（这几行日志本来是调试这个插件时一点点
  加上去的，现在收进一个开关里，不用每次都翻源码改日志级别）。
- `global_max_length`：群/用户没有单独设置字数上限时使用的默认值，`0` = 不限制。

### 强制接管内置主动回复

只要 `active_reply_enable` 为真，插件加载时、以及此后每条群消息处理时，都会检查
AstrBot 主配置里的 `provider_ltm_settings.active_reply.enable`：如果它也是开着的，
会**强制把它写成 `false` 并保存主配置文件**，同时在日志里说明原因，而不是仅仅提示一句
"别同时开"就不管了——两边各自独立判定，一定会导致实际触发频率明显偏离你在 TriggerGuard
里设置的数值，所以直接由 TriggerGuard 统一接管。如果你想恢复用内置的主动回复，把
`active_reply_enable` 关掉即可，插件不会再动那个开关。

## UI 设置页

群/用户的触发区间覆盖、黑名单、屏蔽名单、唤醒关键词，可以在 AstrBot Dashboard 的
`插件 -> TriggerGuard -> Pages -> settings` 里直接管理（表格 + 标签式增删，点击
"保存"即写回磁盘并立即生效）。此页面基于 AstrBot 官方插件 Pages 机制实现
（`context.register_web_api` + `pages/settings/index.html`，与
`astrbot_plugin_livingmemory` 的 Pages 实现方式一致），要求 AstrBot 版本
`4.24.2` 或更高；旧版本没有该能力时，本插件会在日志里提示并回退为"仅配置文件"
模式，核心拦截逻辑不受影响。

左侧侧边栏分两层：上半部分是**管理项导航**（主动回复白名单 / 群聊触发区间 / 用户触发
区间 / 群消息字数上限 / 用户消息字数上限 / 拉黑名单 / 完全屏蔽名单 / 唤醒关键词），点哪个
就只显示哪个面板，不用再在一个页面里滚动着找；下半部分是可折叠的**"协议 / 机器人实例"**
分组（默认收起，点标题展开/收起），里面才是协议筛选列表。这两层是独立的：切换管理项不会
影响当前选中的协议，反过来也一样。

协议列表会通过 `context.platform_manager.platform_insts` 自动识别当前已启动的协议适配器
实例（比如同时跑了两个 aiocqhttp 账号，会分别列出，因为它们的群号/QQ 号命名空间是独立
的，不能混用同一份黑名单）。**`webchat`（AstrBot 自带的网页测试对话）会被过滤掉，不出现
在列表里**——它没有"群"的概念，触发控制对它没有意义。选中某个协议实例后，主动回复白
名单、群/用户触发区间、拉黑、完全屏蔽这几个面板都只显示并编辑该协议下的数据；"添加"
表单也会自动把新条目归到当前选中的协议下（选中"全部"时会显示所有协议的数据并带上协议
列，但此时禁用新增，需要先选一个具体协议才能新增，避免新条目不知道归属哪个协议）。
唤醒关键词是全局生效的，不区分协议，切到这个管理项时协议统计卡片也不会显示。

**"主动回复白名单"面板在最上面，是所有触发区间生效的前提**：`active_reply_enable` 打开
但白名单为空时，插件加载日志会打一条 WARNING 明确提示"当前不会对任何群主动回复"；
群/用户触发区间即使配置了具体数值，只要这个群不在白名单里，`guard_message` 会在真正
判定触发之前就直接跳过（这条跳过日志是 INFO 级别，日志里会带上当前白名单的完整内容，
方便直接对照排查）。

**"群聊触发区间"面板里给一个群配置了区间，会自动把这个群同步进白名单**（UI 里点"添加"
立刻能看到、保存时后端也会做同样的兜底同步，两边双保险），不需要再手动去白名单面板加
一遍——这是从一次真实的踩坑里加的：配了区间但忘了单独加白名单，导致触发区间形同虚设。
反过来不会自动生效：只在白名单面板里加群、不配区间，这个群会使用全局默认区间正常触发；
但如果你**删除**一个群的区间覆盖，这个群不会被自动从白名单里移出（万一你还想让它用默认
区间触发），需要手动去白名单面板删。

选中一个具体协议后，"用户触发区间"（以及"用户消息字数上限"）表单里可以勾选
"全局（不限群）"：这样设置的值不绑定任何一个群，该用户在这个协议下的所有群都用这个值，
不用逐群配置一遍。表格里会用"全部群"标签区分这类条目和普通的按群配置条目，两者可以同时
存在（按群的优先级更高，见上面"解决的问题"里的优先级说明）。

"群消息字数上限" / "用户消息字数上限"这两个面板跟触发区间是完全独立的两套优先级链：
字数超限是"完全屏蔽"式的硬拦截（不进上下文、不触发 LLM），跟这条消息本身有没有触发主动
回复、有没有被直接 @ 都没关系——只要超过对应作用域解析出来的字数上限，就会被挡下。表格
里数字输入框填 `0` 或留空表示"这个作用域不限制"，会覆盖掉更低优先级（群/全局默认）的
限制，不是"没配置"。

有群号的地方（群触发区间、按群配置的用户触发区间）都会在数字后面追加群名，如
`123456789 (示例群名)`——名字来自下面"协议统计"里拉到的群列表，同样只有支持自动获取
群列表的协议才会有，其余协议只显示裸群号。

两张表格最右边还有一列"触发进度"，显示这个作用域当前的**剩余触发条数 / 预计触发条数**
（比如"剩余 12 / 预计 80"表示这一轮还需要 12 条普通消息才会触发一次，本轮随机到的目标
是 80）。这直接读取插件运行时的计数器/随机目标值内存状态，是加载那一刻的快照，不会
随着新消息实时刷新——点"重新加载"可以拿到最新的。从没人在这个作用域说过话、或者进程
刚重启还没积累计数的条目会显示"尚未开始计数"，这不是 bug，是这个作用域确实还没有节奏
状态可展示。

选中协议后，页面顶部还会显示一张"协议统计"卡片，尝试获取：机器人 ID、已加入群聊数、
群成员合计。**目前只有 aiocqhttp（OneBot v11）支持自动获取**——AstrBot 核心没有跨协议
统一的"列出群聊"接口，其他协议（Discord/Telegram/企业微信等）SDK 差异太大，有些甚至
不支持机器人主动查询自己加入了哪些群（比如 Telegram Bot API）。不支持时会如实提示，
不会伪造数据。支持时还会顺带把已加入的群喂给"群号"输入框做自动补全（浏览器原生
datalist，输入群号时可以从下拉列表里选，也可以直接手打不在列表里的群号）。"群成员合计"
是跨群简单求和，不是去重后的用户数（同一个人在多个群里会被重复计入）。

**QQ 号也不用死记硬背了**：侧边栏协议列表下面有个"群成员速查"下拉框（跟"协议统计"一样，
只有支持的协议、且选中了具体协议实例时才会出现），选一个群之后，页面里所有"QQ 号"输入框
（拉黑名单、完全屏蔽名单、用户触发区间、用户消息字数上限）都会自动补全这个群的成员，
下拉里显示"QQ号 — 群名片/昵称"。另外，"用户触发区间"和"用户消息字数上限"这两个面板自己
就带群号输入框，填完群号并失焦后，也会自动按这个群刷新补全列表，不用特地去侧边栏选一次。
群成员数据是按需拉取并缓存的（不会打开设置页就把所有群的成员一次性拉完，群可能有几百上
千人，一次性拉代价太大），点"重新加载"会清空缓存重新拉。同样目前只有 aiocqhttp 支持，
其他协议下拉列表会保持空，只能手动输入。

### 机器人配置路由（跟 TriggerGuard 自己的数据无关，是 AstrBot 核心功能的另一套入口）

最后一个管理项"机器人配置路由"跟前面几个不是一回事：它不读写 `overrides.json`，而是直接
操作 AstrBot 核心的 `UmopConfigRouter`（`context.astrbot_config_mgr.ucr`）——**跟 AstrBot
官方 Dashboard "机器人" 页面编辑窗口里那张 UMO -> 机器人配置 表，是完全同一份数据、同一个
单例对象**，两边改了对方立刻能看到，不存在数据不同步的问题。之所以在这里也做一份，是因为
官方那张表藏在"编辑平台适配器"弹窗里、还要自己拼群号，不够直观；这里选群、选配置，点一下
立即生效，不需要点页面下方全局的"保存"按钮（那个按钮只管 `overrides.json` 里的数据）。

**这里特别澄清一点，避免误解**：一个群没有出现在这张路由表里，**不代表它"没配置好"或者
"不会生效"**——没有单独绑定就是在用默认机器人配置，这是绝大多数群的正常状态，跟本插件的
"主动回复白名单"、触发区间等功能是否生效**没有任何关系**（两者读写的是完全不同的数据）。
如果你遇到"白名单里配了群但主动回复就是不触发"，根源更可能是 AstrBot 的"会话管理"里那个
`session_enabled` 开关被关掉了——那是另一套独立的存储，跟这里的配置路由、也跟 TriggerGuard
自己的数据都不是一回事，本插件目前还没有对接那个开关的读取/提醒（如果这个开关很常被误关，
之后可以再加一个诊断）。

不方便用 UI 时，也可以直接编辑插件数据目录下的 `overrides.json`
（路径类似 `data/plugin_data/astrbot_plugin_trigger_guard/overrides.json`），
下一条消息到达时会按文件修改时间自动重新加载，无需重启：

```json
{
  "active_reply_whitelist": ["aiocqhttp:bot1:123456789"],
  "group_interval": { "aiocqhttp:bot1:123456789": { "min": 50, "max": 100 } },
  "user_interval": { "aiocqhttp:bot1:123456789:10001": { "min": 20, "max": 40 } },
  "user_interval_global": { "aiocqhttp:bot1:10001": { "min": 30, "max": 60 } },
  "group_max_length": { "aiocqhttp:bot1:123456789": 500 },
  "user_max_length": { "aiocqhttp:bot1:123456789:10001": 2000 },
  "user_max_length_global": { "aiocqhttp:bot1:10001": 1000 },
  "blacklist": ["aiocqhttp:bot1:10002"],
  "blocklist": ["aiocqhttp:bot1:10003"],
  "wake_keywords": ["帮我", "在吗"]
}
```

- `active_reply_whitelist` 条目是 `协议实例ID:群号`——群不在这里，`group_interval`/
  `user_interval`/`user_interval_global` 配了也不会生效，这个数组为空就是"完全不主动回复"。
- `group_interval` 的 key 是 `协议实例ID:群号`；`user_interval` 是
  `协议实例ID:群号:QQ号`（本群生效）；`user_interval_global` 是 `协议实例ID:QQ号`
  （跨群生效，同一用户如果两边都配了，`user_interval` 优先）。每个值都是
  `{"min": 最少条数, "max": 最多条数}`，`max` 必须 >= `min`，两者都是 >= 1 的整数。
  协议实例 ID 就是 `event.get_platform_id()` 的值，也是 UI 侧边栏每个协议条目下面
  那行小字，对应 AstrBot 平台配置里那个适配器实例自己的 ID（不是 aiocqhttp/discord
  这种类型名——同类型可以同时跑多个实例）。
- `group_max_length` / `user_max_length` / `user_max_length_global` 的 key 格式跟上面
  三个触发区间字段一一对应，只是值从 `{min,max}` 换成了一个非负整数（字数上限，`0` =
  该作用域明确不限制）。
- `blacklist` / `blocklist` 条目是 `协议实例ID:QQ号`，按协议实例 + 用户全局生效（不区分群）。
- `wake_keywords` 为空数组时不启用关键词唤醒，全局生效不分协议。
- **旧版本数据迁移**：
  - 无协议前缀的 key/条目会自动归入一个特殊的 `_legacy` 协议桶（在 UI 侧边栏显示为
    "旧数据"），不会丢失，但也不会生效在任何真实协议上——需要在 UI 里手动挪到正确的
    协议实例下。
  - 更早版本里 `group_probability`/`user_probability`/`user_probability_global`
    存的是 0-1 浮点概率，加载时会按 `均值 ≈ 1/概率` 近似换算成 `[min, max]` 区间
    （只是粗略换算，不是精确等价），建议迁移后去 UI 里确认/调整一遍具体数值。

触发节奏的计数器（当前攒了多少条、这一轮的随机目标是多少）是纯内存状态，不写入
`overrides.json`，插件重启后会重新开始计数——这是预期行为，只是节奏状态，不是配置数据。

## 已知局限

- 对于**拉黑**用户，如果其消息本身就用了唤醒前缀 / @，AstrBot 仍会走到构建 LLM 请求
  的那一步（打字状态、上下文组装等准备工作会发生），只是在真正调用 Provider 前被
  拦截，不会产生实际的模型调用与回复。这是纯插件方案下能做到的最干净的拦截点
  （`EventType.OnLLMRequestEvent`），如果要连准备工作都跳过，需要修改 AstrBot 核心
  的 `WakingCheckStage` / `RateLimitStage`。
- **完全屏蔽**依赖本插件的消息处理器（`priority=5000`）先于内置
  `group_chat_context` 的 `on_message`（默认优先级 0）执行——这是 AstrBot Star
  handler 优先级机制的正常用法，不是取巧的时序依赖。
- **强制接管**只在插件加载时和之后每条群消息处理时检查一次内置开关，不是实时监听
  配置文件变化；如果 AstrBot 主配置是被外部工具直接改的文件（而不是通过 Dashboard），
  要等下一条群消息到达才会被重新接管。
