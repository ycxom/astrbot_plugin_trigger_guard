# Changelog

本项目的版本号遵循 [Semantic Versioning](https://semver.org/lang/zh-CN/)，格式参考
[Keep a Changelog](https://keepachangelog.com/zh-CN/1.0.0/)。

## [1.0.3] - 2026-07-13

### Added

- **戳一戳（Poke）不再计入触发计数**：AstrBot 里戳一戳跟普通消息走同一条管道
  （`MessageType.GROUP_MESSAGE`，`message_str` 为空字符串），之前会被当成一条
  "普通消息"计入触发计数器；现在会识别消息里的 `Poke` 组件并跳过计数，逻辑上跟
  "机器人自身消息不计入计数"是同一类修复。拉黑/完全屏蔽名单仍然正常拦截戳一戳
  （判定顺序在这条修复之前），不受影响。

## [1.0.2] - 2026-07-08

### Added

- **机器人配置路由管理**：新增"机器人配置路由"管理项，可以给指定的群单独绑定一份
  "机器人配置"（人设/Provider 等）——直接操作 AstrBot 核心的 `UmopConfigRouter`
  （`context.astrbot_config_mgr.ucr`），跟官方 Dashboard "机器人"页面编辑窗口里的
  UMO → 机器人配置表是同一份数据，双向实时同步。选群、选配置，立即生效，不需要经过
  页面下方全局的"保存"按钮。绑定时会自动检查目标配置是否开启了
  `platform_settings.enable_id_white_list`：开了的话，顺手把这个群也追加进那份配置
  自己的 `id_whitelist` 并保存，避免"路由绑对了，但那份配置的白名单没跟着改，最后
  还是不生效"这种两个设置项本该配套却少配一个的坑。

## [1.0.1] - 2026-07-07

### Added

- **UI 设置页**：基于 AstrBot 官方插件 Pages 机制（`context.register_web_api` +
  `pages/settings/index.html`）新增可视化设置页，插件 -> TriggerGuard -> Pages ->
  settings 即可访问，无需再手动编辑 `overrides.json`。旧版 AstrBot 没有该能力时自动
  回退为"仅配置文件"模式，核心拦截逻辑不受影响。
- **协议实例识别**：所有触发相关数据（除 `wake_keywords` 外）按协议实例 ID
  （`event.get_platform_id()`）分区，支持同一协议类型同时跑多个机器人实例而不互相
  污染黑名单/白名单/触发区间。UI 侧边栏可按协议筛选查看/编辑；`webchat` 因为没有
  "群"的概念被主动排除在协议列表外。
- **主动回复触发区间（替代原来的 0-1 概率）**：不再是"每条消息独立掷骰子"的伯努利
  概率模型，改为给每个作用域配置 `[最少, 最多]` 条消息的区间，攒够随机抽到的目标
  条数后触发一次，波动比纯概率模型小，也更符合"这个群大概每 N~M 条消息主动说一次
  话"的直觉。支持 本群用户 > 全局用户(跨群) > 群 > 全局默认 四级优先级覆盖。
- **主动回复白名单（群级别，opt-in）**：`active_reply_enable` 打开后默认不对任何群
  主动回复，只有加进白名单的群才会跑触发区间计数，避免开关一开就对机器人所在的所有
  群生效。给一个群配置了触发区间会自动同步进白名单，无需再手动操作两遍。
- **群 / 用户消息字数上限**：超过上限的消息按"完全屏蔽"处理——不进上下文、不触发
  LLM（包括直接 @ 唤醒），防止超长消息污染人设/上下文。用户设置优先级高于群设置。
- **触发进度展示**：UI 表格显示每个作用域当前的"剩余触发条数 / 预计触发条数"快照。
- **协议统计**：尝试获取机器人 ID、已加入群聊数、群成员合计，并为"群号"输入框提供
  自动补全（目前仅 aiocqhttp / OneBot v11 支持，其他协议如实提示不支持，不伪造数据）。
- **`debug_log` 详细日志开关**：打开后每条群消息的触发判定过程（是否命中黑/白名单、
  当前计数/阈值等）以 INFO 级别打印，不用改 AstrBot 全局日志级别即可排查"配了但不
  触发"一类问题；关闭时降级为 DEBUG，避免大群刷屏。
- **强制接管 AstrBot 内置主动回复**：`active_reply_enable` 打开时，自动检测并强制
  关闭 `provider_ltm_settings.active_reply.enable`（写回主配置文件），避免两边各自
  独立判定导致实际触发频率明显偏离配置数值。
- **群成员速查**：选一个群后，页面里所有"QQ 号"输入框（拉黑名单、完全屏蔽名单、用户
  触发区间、用户消息字数上限）自动补全该群成员（QQ 号 + 群名片/昵称）；"用户触发区间"
  "用户消息字数上限"两个面板自带的群号输入框填完后也会自动刷新补全列表。按需按群拉取
  并缓存，不会一次性拉全部群的成员；目前仅 aiocqhttp 支持。

### Fixed

- **机器人自身消息不再计入触发计数**：修复了机器人自己发送的消息在某些协议/配置下
  会被当作一条"普通消息"计入触发计数器的问题——区间设得很小时（如 min=max=1）会导致
  机器人自己跟自己无限触发。
- 黑名单/完全屏蔽名单用户的消息本就不会计入触发计数（在 `_should_trigger` 判定之前
  就已 return），这次在代码里补充了明确的文档说明，防止后续重构时改乱顺序。

### Changed

- 侧边栏重构为两层导航：上层是管理项（主动回复白名单 / 群聊触发区间 / 用户触发区间 /
  群消息字数上限 / 用户消息字数上限 / 拉黑名单 / 完全屏蔽名单 / 唤醒关键词），点哪个
  显示哪个面板；下层是可折叠的"协议 / 机器人实例"分组，两者相互独立。
- 旧版本（无协议前缀、且概率是 0-1 浮点数）的数据会在加载时自动迁移：无协议前缀的
  归入 `_legacy` 协议桶，浮点概率按 `均值 ≈ 1/概率` 近似换算成 `[min, max]` 区间。

## [1.0.0] - 初始版本

- 独立的群聊 / 用户触发概率覆盖。
- **拉黑**：保留用户聊天记录（正常进入群聊上下文感知），但该用户不能触发 LLM。
- **完全屏蔽**：用户消息不进入任何上下文，也不能触发 LLM。
- **关键词包含唤醒**：消息中包含配置的关键词即可唤醒，不要求前缀或 @。
- 在 `EventType.OnLLMRequestEvent`（请求 LLM 前的最终闸口）统一拦截，保证黑名单/
  完全屏蔽名单在任何触发路径下都生效。
